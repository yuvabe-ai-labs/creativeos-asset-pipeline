# KB System — Architecture & Component Map

**Date:** 2026-06-06
**Topic:** How the Client Knowledge Base (KB) is structured, extracted, and reviewed

---

## What is a KB?

A Client KB is a structured brand intelligence document attached to a client. It is built once
(and re-built whenever source files change) and injected into generation prompts at runtime so
every image/video output stays on-brand without the user re-entering context each time.

**Pipeline:**

```
Documents + Images
      ↓  (upload)
  kb_documents / brand_images tables (Supabase Storage)
      ↓  (AI extraction — GPT-4o)
  TraceableBrandKB JSON (kb_versions table)
      ↓  (field-by-field review)
  status: "in_review" → user approves/edits/rejects each field
      ↓  (mark ready)
  status: "ready" — KB is locked and injectable into prompts
```

Each field in the extracted JSON has shape `{ value, status, source_quote }`:
- `value` — extracted text (or `null` if not found in source material)
- `status` — `"needs_review" | "approved" | "rejected" | "edited"`
- `source_quote` — sentence from the source doc that justified the extraction

## The 7 Modules

| Key | Label | What it captures |
|---|---|---|
| `brand_voice` | Brand Voice | name, tagline, positioning, mission, personality, tone |
| `visual_identity` | Visual Identity | aesthetic, colour palettes, typography, photography style |
| `image_analysis` | Image Analysis | dominant colours, composition, lighting character from brand images |
| `audience_casting` | Audience & Casting | subjects, age range, gender, lifestyle, pain points, desires |
| `image_direction` | Image Direction | shot style, composition, environment, feel |
| `video_direction` | Video Direction | motion style, camera movement, transitions, pacing, music |
| `compliance` | Compliance Rules | preferred verbs/phrases, never-use words/claims/tone, disclaimers |

Module keys and field labels live in `src/lib/kb/constants.ts`.
The full Zod schema is `src/lib/kb/schema.ts` — that is the canonical field list.

## Component Map

```
src/components/kb/
  kb-onboarding-upload-step.tsx   ← Step 1: upload docs/images, trigger extraction
  kb-onboarding-review-step.tsx   ← Step 2: field review shell (state + all handlers)
  kb-module-card.tsx              ← Per-module accordion card with progress bar
  kb-field-row.tsx                ← Single field: edit / approve / reject / re-analyze
  kb-source-panel.tsx             ← Documents & images manager (drawer body)
  kb-re-extract-overlay.tsx       ← Full-screen loading overlay during re-extraction
```

### Responsibility split

| Component | What it owns |
|---|---|
| `kb-onboarding-upload-step.tsx` | File collection UI, calls `/api/clients/[id]/kb/extract`, shows `KBReExtractOverlay` while `useTransition` is pending |
| `kb-onboarding-review-step.tsx` | All review state (`localKB`, `staged`, upload flags), all handlers (`patchField`, `handleEdit`, `handleReject`, `handleApprove`, `handleApproveAll`, `handleReanalyzeField`, `handleMarkReady`, `fireReExtract`, upload + staged-change handlers) |
| `kb-module-card.tsx` | Renders one module's fields via `KBFieldRow`, computes module-level status + progress bar |
| `kb-field-row.tsx` | Renders one field row: value display, edit textarea, approve/reject/undo buttons. Approve-only (no reject) for null-value fields |
| `kb-source-panel.tsx` | Tab panel (Documents / Images) inside the side drawer. Always rendered — not gated on edit mode |
| `kb-re-extract-overlay.tsx` | `visible: boolean` prop only. `bg-background/95 backdrop-blur-md` so background content is hidden |

## Ready Gate

A KB version can only be marked ready when **every field** (including null-value fields) has
`status !== "needs_review"`. This is enforced by `computeReadyStatus` in `src/lib/kb/fill-rate.ts`.

Null is a valid extraction result (source material didn't mention that field). The user must
explicitly approve null fields before marking the KB ready — the system does not auto-approve them.

---

## State Reset on New Version

`KBOnboardingReviewStep` is mounted with `key={activeKBVersion!.id}` in
`src/app/clients/[id]/kb/page.tsx`. When a re-extraction completes and `router.refresh()` loads
a new version, React remounts the component from scratch — all `useState` re-initialize from
the new props. This replaces the previous `useEffect([versionId]) { setState(...) }` pattern that
triggered cascading-renders lint errors.
